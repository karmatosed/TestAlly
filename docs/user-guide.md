# TestAlly User Guide

Version: 1.0.0
Last updated: 2026-03-13

---

## 1. Introduction

TestAlly is an AI-powered accessibility assistant that helps developers perform manual accessibility testing on their components. It combines automated static analysis with AI-generated, WCAG-backed manual testing walkthroughs so you can catch accessibility issues before they reach a formal audit.

### 1.1 Who Is This For?

- **Developers** who want to verify their components are accessible before shipping
- **QA testers** who need structured manual test scenarios for accessibility
- **Designers** who want to understand accessibility implications of their designs
- **OSS maintainers** who want to ensure components meet accessibility standards

### 1.2 What TestAlly Does

1. Accepts your component code (HTML, JSX, CSS, JavaScript)
2. Runs automated accessibility checks (axe-core, ESLint)
3. Analyzes your component's interactive patterns
4. Generates step-by-step manual testing instructions with WCAG citations
5. Provides a confidence score validated by a second AI model

### 1.3 What TestAlly Does Not Do

- Execute or render your code in a browser (static analysis only)
- Automatically fix accessibility issues for you
- Replace a comprehensive accessibility audit
- Test third-party components loaded from external APIs

## 2. Getting Started

### 2.1 Prerequisites

- A modern web browser (Chrome, Firefox, Safari, Edge)
- Component source code you want to test

### 2.2 Accessing TestAlly

Navigate to the TestAlly web application in your browser. No account or login is required for the MVP.

### 2.3 Your First Analysis

1. Open TestAlly in your browser
2. Paste your component's HTML into the code editor
3. If your component has CSS, paste it into the CSS tab
4. If your component has JavaScript, paste it into the JS tab
5. In the description field, write a short description of your component (e.g., "Navigation tabs with three panels")
6. Click **Analyze**
7. Review the results

## 3. Using the Interface

### 3.1 Code Input Area

The code input area has three tabs:

- **HTML/JSX**: Paste your component markup here. Both plain HTML and JSX syntax are supported.
- **CSS**: Paste any associated styles. This helps TestAlly detect issues like removed focus indicators.
- **JS**: Paste associated JavaScript or event handlers. This helps detect interactive behavior patterns.

You do not need to provide all three. At minimum, provide the HTML/JSX markup.

### 3.2 Component Description

The description field accepts a short phrase describing your component's purpose and type. Examples:

- "Accordion with expandable sections"
- "Modal dialog with a close button"
- "Data table with sortable columns"
- "Tab navigation with three panels"
- "Dropdown menu for site navigation"

This description helps TestAlly's AI correctly identify the component pattern and apply the right WCAG criteria.

### 3.3 Analyze Button

Click **Analyze** to submit your component for review. The analysis typically takes a few seconds. You will see a progress indicator while the static analysis and AI generation steps run.

## 4. Understanding Results

### 4.1 Results Overview

After analysis, results are displayed in three sections:

1. **Automated Findings** - Issues found by static analysis tools
2. **Manual Testing Walkthrough** - Step-by-step instructions you should perform
3. **All Clear / Summary** - Overall assessment and confidence score

### 4.2 Automated Findings

These are issues detected by axe-core, ESLint, and TestAlly's custom rules. Each finding includes:

- **Severity**: Critical, Serious, Moderate, or Minor
- **Description**: What the issue is
- **WCAG Criterion**: The specific success criterion being violated
- **Fix Suggestion**: A minimal, semantic code change to resolve the issue
- **Source**: Link to the official WCAG documentation

### 4.3 Manual Testing Walkthrough

This is TestAlly's core output. Each manual test includes:

- **Title**: A short description of what to test
- **WCAG Criteria**: Which success criteria this test covers
- **Priority**: How important this test is (critical, moderate, minor)
- **Steps**: Ordered actions to perform, each with:
  - **Action**: What to do (e.g., "Press Tab to move focus to the button")
  - **Expected Result**: What should happen if the component is accessible
  - **If Fail**: What it means if the expected result does not occur, and guidance on fixing it
- **Sources**: Links to official WCAG and ARIA documentation

### 4.4 Confidence Score

Each analysis includes a confidence score from 0 to 100, generated by a separate validation AI model. This indicates how confident TestAlly is in its recommendations:

| Score Range | Meaning |
|-------------|---------|
| 90-100 | High confidence. Recommendations are well-supported by WCAG criteria. |
| 70-89 | Good confidence. Most recommendations are well-supported. |
| 50-69 | Moderate confidence. Some recommendations may need additional verification. |
| Below 50 | Low confidence. Treat as preliminary guidance; consult an accessibility specialist. |

### 4.5 All Clear

If TestAlly determines that your component does not require manual testing beyond what automated tools cover, you will see an "All Clear" result with an explanation of why no manual tests are needed.

## 5. Manual Testing Walkthrough: How to Execute

### 5.1 Setting Up Your Test Environment

To perform the manual tests TestAlly recommends:

1. Open your component in a browser
2. Disconnect or disable your mouse/trackpad (for keyboard testing)
3. If testing screen reader behavior, enable your operating system's screen reader:
   - **macOS**: VoiceOver (Cmd + F5)
   - **Windows**: NVDA (free download) or Narrator (Win + Ctrl + Enter)
   - **Linux**: Orca
4. Use browser DevTools to inspect focus states and ARIA attributes as needed

### 5.2 Keyboard Testing

Many of TestAlly's manual tests involve keyboard interaction. Key mappings:

| Key | Common Function |
|-----|----------------|
| Tab | Move focus to next interactive element |
| Shift + Tab | Move focus to previous interactive element |
| Enter | Activate buttons, links, and some controls |
| Space | Activate buttons, toggle checkboxes, select options |
| Arrow keys | Navigate within composite widgets (tabs, menus, radio groups) |
| Escape | Close modals, dismiss popups, cancel operations |
| Home / End | Jump to first/last item in a list or group |

### 5.3 Following the Steps

For each manual test:

1. Read the **Action** step
2. Perform the described action on your component in the browser
3. Compare what happened to the **Expected Result**
4. If the result does not match, consult the **If Fail** guidance for diagnosis and fix direction
5. Proceed to the next step

### 5.4 Documenting Results

For each test, record:

- Pass / Fail / Not Applicable
- Browser and assistive technology used
- Screenshots or notes for any failures

## 6. Custom Rule Alerts

TestAlly includes custom accessibility detectors beyond standard automated tools.

### 6.1 Link-as-Button

**What it detects**: An `<a>` element used as a button (has `onClick` but no meaningful `href`).

**Why it matters**: Links and buttons have different keyboard expectations. Links activate with Enter; buttons activate with both Enter and Space. Screen readers announce them differently, affecting user expectations.

**What to do**: Replace the `<a>` with a `<button>`, or add `role="button"`, `tabindex="0"`, and a keydown handler for Space.

### 6.2 Focus Ring Removal

**What it detects**: CSS that removes the default focus outline (`outline: none` or `outline: 0`) without providing an alternative visible focus style.

**Why it matters**: Keyboard users rely on focus indicators to know which element is active. Removing them without replacement makes the interface unusable for keyboard-only users (WCAG SC 2.4.7).

**What to do**: Add a visible focus style. Options include a custom `outline`, `box-shadow`, `border`, or a combination that meets the 3:1 contrast ratio requirement.

## 7. Common Component Patterns

TestAlly recognizes and provides targeted manual tests for these common UI patterns:

| Pattern | Key Manual Tests |
|---------|-----------------|
| **Accordion** | Keyboard expand/collapse, aria-expanded state, focus management |
| **Tabs** | Arrow key navigation, tab/panel association, roving tabindex |
| **Modal / Dialog** | Focus trap, Escape to close, return focus to trigger |
| **Dropdown Menu** | Arrow key navigation, Escape to close, type-ahead |
| **Data Table** | Header association, sort controls, caption/summary |
| **Form** | Label association, error identification, required field indication |
| **Carousel / Slider** | Pause control, keyboard navigation, live region updates |
| **Tooltip** | Keyboard trigger, dismissibility, timing |
| **Navigation** | Skip link, current page indication, mobile menu behavior |
| **Alert / Notification** | Live region announcement, dismissibility, persistence |

## 8. Tips for Best Results

1. **Provide complete code**: Include all relevant HTML, CSS, and JS. Partial snippets may lead to incomplete analysis.
2. **Be specific in your description**: "Accordion" is good. "Accordion with three sections that auto-collapse when another opens" is better.
3. **Include ARIA attributes**: If your component uses ARIA roles, states, and properties, include them so TestAlly can verify correct usage.
4. **Test with real content**: After following TestAlly's walkthrough on your code, repeat the tests with your actual component running in the browser.
5. **Run tests in multiple browsers**: Accessibility behavior can vary. Test in at least Chrome and Firefox.
6. **Use assistive technology**: If TestAlly recommends screen reader testing, actually use a screen reader rather than relying on visual inspection alone.

## 9. Frequently Asked Questions

**Q: Does TestAlly replace automated testing tools like axe?**
A: No. TestAlly uses axe-core as part of its analysis pipeline. It adds value by generating the manual tests that automated tools cannot perform.

**Q: Can I test React/Vue/Angular components?**
A: For the MVP, paste the rendered HTML output of your component along with any relevant JSX source. Framework-specific adapters are planned for future releases.

**Q: Is my code stored or shared?**
A: No. Code submitted to TestAlly is used for analysis only and is not persisted or shared. See the deployment manual for self-hosting options if data privacy is a concern.

**Q: What if TestAlly says "All Clear" but an auditor still finds issues?**
A: TestAlly provides guidance based on known patterns and WCAG criteria, but it is not a substitute for a professional accessibility audit. The confidence score indicates the reliability of the assessment.

**Q: Can I use TestAlly in CI/CD?**
A: CI/CD integration via GitHub Actions is a stretch goal. When available, TestAlly will be able to post manual test scenarios as comments on pull requests.

**Q: What WCAG version does TestAlly target?**
A: WCAG 2.2 Level AA. The architecture is designed to accommodate future updates including WCAG 3.0.
